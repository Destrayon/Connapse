# Connapse v0.3.0 — Connector Architecture, Cloud Storage & Agentic Search

## Context

v0.2.0 delivered full security (Cookie + PAT + JWT), RBAC, agent management, audit logging, invite-only registration, CLI auth/distribution, and 256 passing tests. The platform is secure and multi-user.

v0.3.0 generalizes the storage model into a pluggable connector architecture, adds cloud storage backends (AWS S3, Azure Blob), introduces per-user cloud identity for IAM-derived scope enforcement, formalizes the LLM layer, and ships agentic iterative search.

---

## Core Decisions

### Naming
- **Container** = the storage unit in Connapse (an instance of knowledge, indexed and searchable). The term, API paths, and UI stay unchanged.
- **Connector** = the technology type backing a container (MinIO, Filesystem, InMemory, S3, AzureBlob).

### Connector Types

| Type | Description | Live Watch |
|------|-------------|------------|
| `MinIO` | Existing S3-compatible storage via global StorageSettings | No |
| `Filesystem` | Watched local folder (Windows/Linux/macOS via FileSystemWatcher) | Yes |
| `InMemory` | Ephemeral — cleaned on startup. Dynamic short-term RAG / agent working memory | No |
| `S3` | AWS S3 bucket. IAM roles / DefaultAWSCredentials only. No stored keys. | No (sync-on-demand in v0.3.0) |
| `AzureBlob` | Azure Blob Storage via managed identity / DefaultAzureCredential | No (sync-on-demand in v0.3.0) |

### Security Model

**Cloud connectors (S3, AzureBlob)**:
- The connector's service credential (IAM role, managed identity) defines the ceiling of what Connapse can index from the cloud — Connapse only sees what those credentials permit.
- Each Connapse user can link a cloud identity per provider (one AWS identity, one Azure identity). These are obtained through provider auth flows and stored encrypted on the user's profile.
- When a user accesses a cloud container, Connapse uses their linked cloud identity to discover their accessible prefixes via the cloud's IAM (AWS `SimulatePrincipalPolicy` / S3 prefix enumeration via STS, Azure RBAC API). The cloud's IAM is the source of truth — Connapse maintains no shadow permission table for cloud connectors.
- Discovered scopes are cached with a TTL (15 minutes) and enforced at query time and in the file browser.
- Users with no linked cloud identity cannot access cloud containers (clear error: "Link your AWS identity to access this container").

**Local connectors (MinIO, Filesystem, InMemory)**:
- Role-level RBAC (current behavior). All users of the appropriate Connapse role can access all local containers.
- Per-connector granular ACLs deferred to v0.4.0.

### Per-Container Settings

Each container can override the global settings (chunking strategy, embedding model, search defaults, file type filters). The resolution order:

```
appsettings.json → env vars → global DB settings → container settings (highest priority)
```

`IContainerSettingsResolver` handles the merge transparently. Services use this instead of `IOptionsMonitor<T>` directly when container context is available.

Changing a container's embedding model triggers a reindex warning — existing chunks were embedded with a different model and are no longer comparable.

### RS256

Implemented but optional. HS256 remains the default. When RS256 is enabled in settings, Connapse generates a 2048-bit RSA key pair, exposes `/.well-known/openid-configuration` and `/.well-known/jwks.json`, and shows step-by-step setup guidance for AWS OIDC trust relationships in the UI.

RS256 is a prerequisite for AWS OIDC federation (per-user scope discovery for S3 containers). The Cloud Identities section of the user profile shows a warning if the user tries to connect an AWS identity when RS256 is not enabled.

### Agentic Search

New `SearchMode.Agentic` — an LLM-driven iterative retrieval loop:
1. Generate initial search queries from the user's question
2. Execute searches via the existing hybrid search pipeline
3. LLM evaluates whether retrieved chunks are sufficient
4. If not satisfied and under max iterations: refine queries, repeat
5. Return all collected chunks, deduplicated and ranked

Disabled in the UI unless an LLM provider is configured. Max iterations configurable (default 3).

## Phase Details

### Session A — IConnector Abstraction + Schema Migration

#### Phase 1: Core Interfaces (Connapse.Core)

```csharp
public enum ConnectorType { MinIO, Filesystem, InMemory, S3, AzureBlob }

public interface IConnector
{
    ConnectorType Type { get; }

public record ConnectorFileEvent(ConnectorFileEventType EventType, string Path, string? OldPath = null);

public interface IConnectorFactory

    IConnector Create(ContainerEntity container);
}

public interface IContainerSettingsResolver
{
    Task<ChunkingSettings> GetChunkingSettingsAsync(Guid containerId, CancellationToken ct = default);
    Task<EmbeddingSettings> GetEmbeddingSettingsAsync(Guid containerId, CancellationToken ct = default);

public record ContainerSettingsOverrides

    public ChunkingSettings? Chunking { get; set; }

```

#### Phase 2: Schema Migration + API

**EF migration** — additions to `containers` table:
- `connector_type` (integer, NOT NULL, default 0 = MinIO) — all existing containers become `MinIO`
- `connector_config` (JSONB, nullable) — type-specific configuration blob
- `is_ephemeral` (boolean, NOT NULL, default false) — true for InMemory containers
- `settings_overrides` (JSONB, nullable) — per-container settings, null = use global

**`ContainerSettingsResolver`** (Connapse.Storage):
- Loads container from DB, deserializes `settings_overrides`, merges with `IOptionsMonitor<T>` global values
- Returns effective settings — callers don't need to know whether overrides exist

**New API endpoints**:
- `GET /api/containers/{id}/settings` — get container settings overrides
- `PUT /api/containers/{id}/settings` — save container settings overrides (RequireEditor)

**Update `Container` model** to expose `ConnectorType`, `ConnectorConfig` (as typed object), `SettingsOverrides`.

**Update container creation** to accept `connectorType` in request body (defaults to `MinIO`).

**Container settings UI** — new "Settings" tab on the container detail page with per-category override forms. Shows global default as placeholder when no override exists.

**State after Session A**: All 12 projects build. Existing MinIO containers work identically. Per-container settings can be saved and loaded. No connector behavior changes yet.

### Session B — MinIO Connector + Filesystem Connector

#### Phase 3: MinIO as IConnector

**`MinioConnector`** (Connapse.Storage):
- Implements `IConnector` with `Type = ConnectorType.MinIO`, `SupportsLiveWatch = false`
- Wraps existing MinIO/S3 client (`AWSSDK.S3`)
- Uses global `StorageSettings` — no per-container MinIO server config
- `WatchAsync` throws `NotSupportedException`

**`ConnectorFactory`** (Connapse.Storage):
- MinIO: creates `MinioConnector` using global `StorageSettings`
- Filesystem/InMemory/S3/AzureBlob: resolved in later sessions — throws `NotImplementedException` until then

**`IngestionPipeline` update**: reads files via `IConnector` (resolved from `IConnectorFactory` by containerId) instead of directly via `IKnowledgeFileSystem`. `IKnowledgeFileSystem` retained but ingestion moves to the connector abstraction.

#### Phase 4: Filesystem Connector

**`FilesystemConnector`** (Connapse.Storage):
- `Type = ConnectorType.Filesystem`, `SupportsLiveWatch = true`
- `connector_config` schema: `{ "rootPath": "/data/docs", "includePatterns": ["*.pdf","*.md"], "excludePatterns": ["*.tmp","~*"] }`
- `ReadFileAsync`, `WriteFileAsync`, `DeleteFileAsync` via `System.IO`
- `ListFilesAsync`: recursive `Directory.EnumerateFiles` filtered by include/exclude patterns
- `WatchAsync`: wraps `FileSystemWatcher`, yields `ConnectorFileEvent` per OS event
- Cross-platform path normalization (forward slashes internally, OS-specific on disk)

**`ConnectorWatcherService`** (Connapse.Web, BackgroundService):
- On startup: loads all `ConnectorType.Filesystem` containers, starts one `FileSystemWatcher` per container
- Event debounce: 750ms — coalesces rapid changes to the same path before acting
- `Created`/`Changed`: enqueue ingestion job (file read via `FilesystemConnector`)
- `Deleted`: find document by full path, delete from DB + vector store (cascade)
- `Renamed`: delete old path document, enqueue ingestion for new path
- Buffer overflow fallback: full rescan every 5 minutes (OS event buffer can overflow under heavy churn)
- Locked file retry: exponential backoff, 3 attempts (2s / 4s / 8s), then log warning and skip
- New container created at runtime: watcher started immediately, initial sync enqueued

**Initial sync** on container creation:
- Enumerate all files matching patterns under root path
- Enqueue ingestion job per file
- Progress visible via existing SignalR batch progress

**Container creation UI update**: connector type selector. When `Filesystem` selected, shows root path field + include/exclude pattern inputs with platform-appropriate placeholder (e.g., `C:\Documents\Wiki` on Windows, `/data/wiki` on Linux).

**State after Session B**: Filesystem containers auto-ingest on creation, react to file system events in real time, handle deletions and renames. MinIO containers use `IConnector` abstraction. The user never sees the watch process.

### Session C — InMemory + S3 + Azure Blob Connectors

#### Phase 5: InMemory Connector

**`InMemoryConnector`** (Connapse.Storage):
- `Type = ConnectorType.InMemory`, `SupportsLiveWatch = false`
- Files stored as `ConcurrentDictionary<string, (byte[] Content, string? ContentType)>`
- Registered as a keyed singleton per containerId
- Files are gone when the process exits — this is by design

**Startup cleanup** (Program.cs):
- On app start, delete all containers where `is_ephemeral = true` (cascade deletes chunks, vectors, documents)
- InMemory containers are recreated fresh each run

**`IConnectorFactory`** updated: returns the in-process `InMemoryConnector` instance for `ConnectorType.InMemory`.

Design note: chunks and embeddings for InMemory containers still write to PostgreSQL during the session (reuses the full ingestion pipeline). Cleanup on startup removes them. This gives full search quality without any new infrastructure. The "in-memory" experience to the user is that content disappears on restart.

#### Phase 6: S3 + Azure Blob Connectors

**`S3Connector`** (Connapse.Storage):
- `Type = ConnectorType.S3`, `SupportsLiveWatch = false`
- `connector_config`: `{ "bucketName": "company-docs", "region": "us-east-1", "roleArn": "arn:aws:iam::123:role/ConnapseIndexer" }` (`roleArn` optional — uses instance profile / env vars if absent)
- Uses `Amazon.S3.AmazonS3Client` with `Amazon.Runtime.DefaultAWSCredentials` — no stored keys ever
- If `roleArn` provided: calls `STS:AssumeRole` to obtain temporary credentials for that role
- Sync-on-demand: `POST /api/containers/{id}/sync` enumerates bucket, compares content hashes, ingests new/changed, deletes removed

**`AzureBlobConnector`** (Connapse.Storage):
- `Type = ConnectorType.AzureBlob`, `SupportsLiveWatch = false`
- `connector_config`: `{ "storageAccountName": "myaccount", "containerName": "docs", "managedIdentityClientId": "optional-client-id" }`
- Uses `Azure.Storage.Blobs.BlobServiceClient` with `Azure.Identity.DefaultAzureCredential`
- Sync-on-demand: same `POST /api/containers/{id}/sync` endpoint

**`S3ConnectionTester`** + **`AzureConnectionTester`** implementing `IConnectionTester`:
- Test by listing bucket/container root with provided credentials
- Returns connected/error + storage account details

**Container creation UI**: S3 and AzureBlob options with their respective config fields. "Test Connection" button per existing pattern.

**New sync endpoint**:
- `POST /api/containers/{id}/sync` (RequireEditor) — triggers full rescan for S3/AzureBlob containers
- Returns a batch ID for progress tracking via existing SignalR infrastructure
- Returns 400 for connector types that use live watch (Filesystem) or have no remote (InMemory)

**State after Session C**: All 5 connector types functional. S3 and Azure Blob containers can be created, configured, and synced on demand. No per-user scope enforcement yet — all users with container access see everything the service credential can see.

### Session D — User Cloud Identities + Auth Flows

#### Phase 7: user_cloud_identities Table + Auth Flows

**New entity** (Connapse.Identity):

```csharp
public class UserCloudIdentityEntity
{
    public Guid Id { get; set; }
    public string UserId { get; set; } = null!;
    public CloudProvider Provider { get; set; }   // AWS | Azure
    public string IdentityDataJson { get; set; } = null!;  // encrypted, provider-specific
    public DateTime CreatedAt { get; set; }
    public DateTime? LastUsedAt { get; set; }
    // No ExpiresAt — identities are confirmed via the provider at discovery time
}

public enum CloudProvider { AWS, Azure }
```

**EF migration**: `user_cloud_identities` table.

**`ICloudIdentityStore`** — CRUD: create, get by user+provider, list by user, delete.

**AWS Identity Flow** (OIDC federation — requires RS256 enabled):
- User: Profile → Cloud Identities → "Connect AWS"
- If RS256 not enabled: show blocking warning with link to settings
- Connapse calls `STS:AssumeRoleWithWebIdentity` using the user's current Connapse JWT
- AWS must already have Connapse configured as a trusted OIDC provider (admin one-time setup)
- On success: stores `{ "principalArn": "...", "accountId": "..." }` — no keys stored, just identity facts
- UI shows: connected AWS account ID + principal ARN

**Azure Identity Flow** (OAuth2/OIDC):
- User: Profile → Cloud Identities → "Connect Azure"
- Standard Azure AD authorization code flow via `GET /.azure/connect` → redirect → `GET /.azure/callback`
- On callback: exchange code for tokens, extract object ID + tenant ID from ID token
- Stores `{ "objectId": "...", "tenantId": "...", "displayName": "..." }` — no access tokens stored
- UI shows: connected Azure account display name + tenant

**Profile page update**: "Cloud Identities" section with per-provider connect/disconnect buttons and connected identity display.

**State after Session D**: Users can link AWS and Azure identities. Stored encrypted. Auth flows work. Cloud containers still show everything the service credential sees.

### Session E — Scope Discovery + Query-Time Enforcement

#### Phase 8: ICloudIdentityProvider + Scope Cache + Enforcement

```csharp
public interface ICloudIdentityProvider
{
    CloudProvider Provider { get; }

    Task<IReadOnlyList<string>> DiscoverScopesAsync(
        string userId,
        ContainerEntity container,
        CancellationToken ct = default);
}

// Scope result: list of path prefixes the user can access
// e.g., ["/engineering/", "/public/"] or ["/"] for full access
// Empty list = no access
```

**`AwsIdentityProvider`** (Connapse.Storage):
- Loads user's linked AWS identity
- Uses STS to get temporary credentials for the user's principal
- Lists accessible S3 prefixes by combining `IAM:SimulatePrincipalPolicy` evaluation with the container's bucket
- Normalizes results to Connapse path prefix format

**`AzureIdentityProvider`** (Connapse.Storage):
- Loads user's linked Azure identity (object ID + tenant ID)
- Calls Azure RBAC API to list role assignments for the user on the storage container
- Maps Azure RBAC roles to accessible blob prefixes
- Normalizes to path prefix format

**`IConnectorScopeCache`** (Connapse.Web, in-memory with IMemoryCache):
- Cache key: `scope:{userId}:{containerId}`
- TTL: 15 minutes (configurable via settings)
- `GetOrDiscoverAsync(userId, container)` — returns cached scopes or calls `ICloudIdentityProvider`

**Enforcement points**:
- `ContainersEndpoints` file listing (`GET /api/containers/{id}/files`): intersect requested path with user's scopes, 403 if outside
- `SearchEndpoints` (`GET|POST /api/containers/{id}/search`): inject user's scopes as path prefix filters into `SearchOptions` before passing to `IKnowledgeSearch`
- `FoldersEndpoints`: scope-check folder paths before listing/creating/deleting
- Only applied for `ConnectorType.S3` and `ConnectorType.AzureBlob` containers

**No-identity handling**: user accesses an S3/Azure container with no linked identity for that provider → HTTP 403 with body:
```json
{ "error": "cloud_identity_required", "message": "Link your AWS identity in Profile → Cloud Identities to access this container." }
```

**State after Session E**: Cloud container queries enforce IAM-derived scopes per user. File browser shows only accessible content. Searches return only chunks within user's allowed prefixes.

### Session F — RS256 + JWKS + Setup Guidance

#### Phase 9: RS256 Implementation

**Settings** (added to `Identity:Jwt` section):
```json
{
  "Identity": {
    "Jwt": {
      "SigningAlgorithm": "HS256",
      "Secret": "...",
      "RsaPrivateKeyPem": ""
    }
  }
}
```

`RsaPrivateKeyPem` overridable via env var `CONNAPSE_RSA_PRIVATE_KEY`. Stored encrypted in DB settings when set via UI.

**When RS256 is enabled**:
- `ITokenService` signs JWTs with RS256 using the configured private key
- `/.well-known/openid-configuration` — OIDC discovery document with issuer, JWKS URI, supported algorithms
- `/.well-known/jwks.json` — public key in JWK format (AWS and Azure read this to verify tokens)
- Existing HS256 tokens continue to validate during a transition window (both verifiers active for 90 minutes)

**Key generation via UI**:
- Settings → Security → JWT Signing section
- "Enable RS256" button: generates 2048-bit RSA key pair, displays public JWKS URL, prompts admin to store private key securely
- "Import Existing Key" option: paste PEM-encoded private key

**Setup guidance shown in UI** (step-by-step, not just a link to docs):
- AWS: how to add Connapse as an OIDC identity provider in IAM Console (provider URL = Connapse base URL, audience = configured client ID)
- Azure: how to configure workload identity federation on a managed identity or app registration

**State after Session F**: RS256 optional and working. JWKS endpoint live. End-to-end AWS OIDC federation functional when RS256 is enabled. HS256 remains the default.

### Session H — ILlmProvider + Agentic Search

#### Phase 11: ILlmProvider Formalization

```csharp
public interface ILlmProvider
{
    string Provider { get; }
    string ModelId { get; }

    Task<string> CompleteAsync(
        string systemPrompt,
        string userPrompt,
        LlmOptions options,
        CancellationToken ct = default);

    IAsyncEnumerable<string> StreamAsync(
        string systemPrompt,
        string userPrompt,
        LlmOptions options,
        CancellationToken ct = default);
}

public record LlmOptions(float Temperature = 0.7f, int MaxTokens = 2048);
```

**`OllamaLlmProvider`**: formalizes the existing Ollama HTTP client pattern used in `CrossEncoderReranker`. Reuses `AddHttpClient<T>()` registration.

**`OpenAiLlmProvider`**: OpenAI chat completions API. Accepts any OpenAI-compatible endpoint (Groq, Together, LM Studio, etc.) via `BaseUrl` override.

**`AnthropicLlmProvider`**: Anthropic Messages API (`claude-*` models).

**LLM Settings UI update**: provider selector (Ollama | OpenAI | Anthropic | AzureOpenAI), model, API key, base URL. Connection test button.

#### Phase 12: Agentic Search Mode

**New `SearchMode.Agentic`** in `SearchMode` enum.

**`AgenticSearchService`** (Connapse.Search), implements `IKnowledgeSearch` for agentic mode:

Loop (up to `MaxIterations`, default 3):
1. LLM generates search queries from the original question (or refined question from previous iteration)
2. Execute searches via existing `HybridSearchService`
3. Deduplicate collected chunks across iterations (by chunk ID)
4. LLM evaluates: "Are these chunks sufficient to answer the question?" Returns yes/no + suggested refinements
5. If no and iterations remain: refine queries, continue loop

Returns standard `SearchResult` with additional `AgenticMetadata`:
```csharp
public record AgenticMetadata(
    int IterationsRun,
    int MaxIterations,
    IReadOnlyList<string> QueriesUsed,
    bool EarlyExit);
```

**Configuration** (LlmSettings):
- `AgenticSearch:MaxIterations` (default 3)
- `AgenticSearch:EvaluationSystemPrompt` (overridable)

**UI**:
- Agentic mode toggle in search — hidden/disabled if no LLM provider is configured
- Progress display via SignalR: "Iteration 2 of 3 — expanding search..." pushed during loop
- Final result shows queries used + iterations run (collapsible metadata)

**API**: `SearchMode.Agentic` accepted in existing `GET|POST /api/containers/{id}/search` endpoints. Returns 400 with message "Agentic search requires an LLM provider to be configured" if no LLM is set up.

**State after Session H**: `SearchMode.Agentic` available across all connector types when LLM is configured. Disabled by default.

## Breaking Changes

### v0.3.0 — Connector Type on Container Creation

**Change**: Container creation accepts optional `connectorType` field. Defaults to `MinIO` — no breaking change for existing API consumers.

**Migration**: All existing containers automatically receive `connector_type = MinIO` via the EF migration. No data loss. No action required.

### v0.3.0 — SearchMode.Agentic Added

**Change**: `SearchMode` enum gains `Agentic` value. Existing `Semantic`, `Keyword`, `Hybrid` unchanged.

**Migration**: No action required. Passing `mode=Agentic` without a configured LLM returns HTTP 400 with a clear error.

## Deferred to v0.4.0

- **Per-connector granular ACLs for local connectors** (Option B RBAC — per-user per-container scope assignments for MinIO/Filesystem)
- **Live events for S3/AzureBlob** (SQS Event Notifications, Azure Event Grid)
- **Cross-container queries** (multi-target `POST /api/search` spanning multiple containers)
- **Additional connectors** (GitHub repos, Notion, Slack)
- **Cohere / Voyage AI embedding providers**
- **Agent session memory** (richer InMemory connector use cases for agent state)
- **Per-user granularity for local connectors**
