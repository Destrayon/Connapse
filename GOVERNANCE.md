# Governance

## Overview

Connapse uses a **BDFL (Benevolent Dictator For Life)** governance model. This is appropriate for the project's current stage — a small team with a single primary maintainer making architectural and release decisions.

As the community grows, governance will evolve to distribute decision-making more broadly.

## Project Lead

**Destrayon** ([@Destrayon](https://github.com/Destrayon)) is the project lead and has final authority on:

- Architectural decisions
- Release timing and versioning
- Roadmap priorities
- Accepting or rejecting contributions

## Roles

### Contributor

Anyone who has had a pull request merged into the project.

**How to become one:** Submit a PR that gets accepted. See the [Contributing Guide](https://github.com/Destrayon/Connapse/blob/main/CONTRIBUTING.md) for guidelines.

**Responsibilities:**
- Follow the [Code of Conduct](https://github.com/Destrayon/Connapse/blob/main/CODE_OF_CONDUCT.md)
- Follow project conventions described in the [Contributing Guide](https://github.com/Destrayon/Connapse/blob/main/CONTRIBUTING.md)

### Collaborator

A contributor who has demonstrated sustained, quality contributions and is granted write access to the repository.

**How to become one:** After multiple accepted PRs that show understanding of the codebase and conventions, the project lead may invite you as a collaborator.

**Additional responsibilities:**
- Review pull requests from other contributors
- Triage and label issues
- Maintain documentation

### Maintainer

A collaborator trusted with release authority and architectural input.

**How to become one:** Invitation from the project lead, based on sustained high-quality contributions, architectural understanding, and alignment with project goals.

**Additional responsibilities:**
- Merge pull requests
- Participate in architectural decisions
- Help shape the roadmap
- Manage releases

## Decision-Making

1. **Day-to-day decisions** (bug fixes, small features, docs) are made by whoever is doing the work, following existing conventions.
2. **Architectural decisions** are documented in the codebase and discussed in GitHub Issues or Discussions before implementation.
3. **Disputes** are resolved by the project lead after considering community input.

## Contributing

See the [Contributing Guide](https://github.com/Destrayon/Connapse/blob/main/CONTRIBUTING.md) for the full contribution process, including development setup, code conventions, and pull request guidelines.

## Changes to Governance

This governance model will be revisited as the project matures. Changes will be proposed via GitHub Discussions and require approval from the project lead.
